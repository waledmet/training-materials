Why Do We Need Forms?

Without forms — hardcoded data only:

post = Post.objects.create(title="Hardcoded", content="Not flexible")

With forms — users provide their own data:

<form method="post">
    <input type="text" name="title">
    <textarea name="content"></textarea>
    <button type="submit">Submit</button>
</form>

Plain HTML vs Django Forms

Plain HTML problems:

  • ❌ No validation
  • ❌ No security (CSRF attacks)
  • ❌ Repetitive error handling
  • ❌ Manual HTML writing

Django Forms solve all of this:

  • ✅ Automatic HTML generation
  • ✅ Built-in validation
  • ✅ CSRF protection
  • ✅ Clean error handling

Creating a Django Form

# forms.py
from django import forms

class ContactForm(forms.Form):
    name = forms.CharField(max_length=100)
    email = forms.EmailField()
    message = forms.CharField(widget=forms.Textarea)

Common field types:

  • CharField - text input
  • EmailField - validates email format
  • IntegerField - numbers only
  • BooleanField - checkboxes

ModelForm - Auto-Generate from Model

# forms.py
from django import forms
from .models import Post

class PostForm(forms.ModelForm):
    class Meta:
        model = Post
        fields = ['title', 'content', 'category']

Why ModelForm?

  • ✅ No duplication — defined once in model
  • ✅ Automatically saves to database
  • ✅ Inherits model validation rules

Handling Forms in Views

def contact(request):
    if request.method == 'POST':
        form = ContactForm(request.POST)
        if form.is_valid():
            # Safe to use the data
            name = form.cleaned_data['name']
            return redirect('success')
    else:
        form = ContactForm()

    return render(request, 'contact.html', {'form': form})

GET → show empty form | POST → validate & save

Rendering Forms in Templates

<form method="post">

    <button type="submit">Submit</button>

Three quick render options:

  • {{ form.as_p }} - wrapped in <p> tags
  • {{ form.as_ul }} - wrapped in <li> tags
  • {{ form.as_table }} - table layout

CSRF Protection

Cross-Site Request Forgery - a security attack

<form method="post">
    {% csrf_token %}   <!-- ALWAYS include this! -->
    ...
</form>
  • Django generates a unique secret token per session
  • Every POST form must include it
  • Django rejects requests without a valid token

Forgetting this = 403 Forbidden error!

Form Validation Levels

  • Built-in: EmailField, IntegerField, required=True
  • Field-level: custom clean_<fieldname>() method
  • Form-level: override clean() for cross-field rules
class ContactForm(forms.Form):
    email = forms.EmailField()

    def clean_email(self):
        email = self.cleaned_data['email']
        if not email.endswith('@example.com'):
            raise forms.ValidationError("Must be a company email!")
        return email

Django's Built-in Auth System

Django ships with everything you need:

  • ✨ User model - username, email, password
  • 🔐 Password hashing - never stored in plain text
  • 🔑 Login / Logout functions
  • 🛡️ Authentication decorators
  • ⚙️ Session management
from django.contrib.auth.models import User
from django.contrib.auth import authenticate, login, logout

Login View

from django.contrib.auth import authenticate, login

def login_view(request):
    if request.method == 'POST':
        username = request.POST['username']
        password = request.POST['password']
        user = authenticate(request, username=username, password=password)
        if user is not None:
            login(request, user)
            return redirect('home')
        else:
            messages.error(request, 'Invalid credentials')
    return render(request, 'login.html')

Logout View

from django.contrib.auth import logout

def logout_view(request):
    logout(request)
    return redirect('login')

Checking login status in templates:

{% if user.is_authenticated %}
    <a href="/logout/">Logout, {{ user.username }}</a>
{% else %}
    <a href="/login/">Login</a>
{% endif %}

User Registration

from django.contrib.auth.models import User

def register_view(request):
    if request.method == 'POST':
        username = request.POST['username']
        password = request.POST['password']
        user = User.objects.create_user(
            username=username,
            password=password
        )
        login(request, user)
        return redirect('home')
    return render(request, 'register.html')

Use create_user() — never create() — it hashes the password!

Built-in UserCreationForm

from django.contrib.auth.forms import UserCreationForm

def register_view(request):
    if request.method == 'POST':
        form = UserCreationForm(request.POST)
        if form.is_valid():
            user = form.save()
            login(request, user)
            return redirect('home')
    else:
        form = UserCreationForm()
    return render(request, 'register.html', {'form': form})

Includes username, password, and confirm password!

Protecting Views with @login_required

from django.contrib.auth.decorators import login_required

@login_required
def create_post(request):
    # Only logged-in users can access this
    pass

@login_required
def dashboard(request):
    posts = Post.objects.filter(author=request.user)
    return render(request, 'dashboard.html', {'posts': posts})

Unauthenticated users are redirected to the login page automatically!

Accessing the Current User

# In views
@login_required
def create_post(request):
    if request.method == 'POST':
        form = PostForm(request.POST)
        if form.is_valid():
            post = form.save(commit=False)
            post.author = request.user   # attach current user
            post.save()
            return redirect('home')

request.user gives you the logged-in user object

Class-Based Views (CBVs)

Write less code for common patterns:

CBVPurpose
ListViewShow list of objects
DetailViewShow single object
CreateViewCreate new object
UpdateViewEdit existing object
DeleteViewDelete an object

CBV vs Function-Based View

# Function-based (verbose)
def post_list(request):
    posts = Post.objects.all()
    return render(request, 'post_list.html', {'posts': posts})

# Class-based (concise)
from django.views.generic import ListView

class PostListView(ListView):
    model = Post
    template_name = 'post_list.html'
    context_object_name = 'posts'

Protecting CBVs

from django.contrib.auth.mixins import LoginRequiredMixin
from django.views.generic import CreateView

class PostCreateView(LoginRequiredMixin, CreateView):
    model = Post
    fields = ['title', 'content']
    template_name = 'post_form.html'
    success_url = '/posts/'

    def form_valid(self, form):
        form.instance.author = self.request.user
        return super().form_valid(form)

Use LoginRequiredMixin instead of @login_required for CBVs

Messages Framework

Give users feedback after actions:

from django.contrib import messages

def create_post(request):
    if form.is_valid():
        form.save()
        messages.success(request, 'Post created successfully!')
        return redirect('home')
    else:
        messages.error(request, 'Please fix the errors below.')
<!-- In base template -->
{% for message in messages %}
    <div class="alert alert-{{ message.tags }}">
        {{ message }}
    </div>
{% endfor %}

Message Levels

messages.debug(request, 'Debug info')
messages.info(request, 'FYI — something happened')
messages.success(request, 'Great! It worked.')
messages.warning(request, 'Be careful!')
messages.error(request, 'Something went wrong.')
  • Maps to Bootstrap alert classes automatically
  • Messages disappear after one page load
  • Perfect for form submission feedback

Security Best Practices

  • 🔒 Always use {% csrf_token %} in POST forms
  • 🔒 Never store plain passwords — use create_user()
  • 🔒 Validate all input — client-side is not enough
  • 🔒 Use @login_required on protected views
  • 🔒 Check object ownership before allowing edits
  • 🔒 Use HTTPS in production

Watch Out For

  • ❌ Forgetting {% csrf_token %} → 403 error
  • ❌ Using User.objects.create() instead of create_user()
  • ❌ Not checking form.is_valid() before using data
  • ❌ Missing @login_required on sensitive views
  • ❌ Not showing validation errors in templates
  • ❌ Forgetting commit=False when setting extra fields

Everyone makes these — learn from them!
